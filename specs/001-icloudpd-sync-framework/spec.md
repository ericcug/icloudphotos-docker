# Feature Specification: iCloud 照片下载器 (iCloud Photo Downloader)

**Feature Branch**: `001-icloudpd-sync-framework`

**Created**: 2026-05-20

**Status**: Draft

**主要目标**: 实现一个将 iCloud 照片下载到本地的 Docker 程序。

**Input**: User description: "导入icloudpd，根据icloudpd的接口开发以下功能：环境与配置管理模块：负责系统启动时的初始化、权限隔离与参数解析。认证与安全模块：维护 iCloud 的登录会话状态，处理 MFA（多因素认证）生命周期。核心同步引擎：执行实际的元数据比对和文件下载。后置处理流水线：照片下载完成后的附加操作。采用流水线/插件式设计最佳。消息通知总线：向用户播报系统事件。远程控制与监听服务：通过外部命令干预同步进程（无需登录宿主机终端）。"



## Overview

本项目的核心目标是将 iCloud 照片库（含照片、视频和 Live Photo）自动同步下载到本地存储。
系统以 Docker 容器方式运行，用户只需提供 iCloud 账号和基本配置即可实现无人值守的媒体备份。
围绕这一核心目标，系统提供六大功能模块：环境配置、认证安全、同步引擎、后置处理、消息通知和远程控制。

## Clarifications

### Session 2026-05-20

- Q: iCloud 云端删除照片时，本地已下载的副本应如何处理？ → A: 提供配置选项，用户可选择保留本地副本或同步删除
- Q: 远程控制接口的安全和通信机制？ → A: 通过 Telegram Bot 实现双向通信控制；若不配置 Telegram 则回退为本地配置文件方式
- Q: 预期的照片库规模上限？ → A: 中型（≤ 50,000 张照片/视频），覆盖典型家庭多年积累的 iCloud 图库
- Q: 后置处理插件的形态（接口契约）？ → A: Python 模块形式，实现约定的接口类，由主进程动态加载并在同进程内调用
- Q: 认证会话的本地存储安全级别？ → A: 文件系统权限保护（chmod 600），不加密内容，依赖宿主机文件系统安全
- Q: Docker 容器中用户如何完成 MFA 验证交互？ → A: 用户在 Telegram Bot 中发送验证码，系统通过 Bot 接收并完成 MFA 认证
- Q: 同步进程崩溃后的恢复机制？ → A: 自动重启同步（指数退避 1min→5min→15min），最多重试 3 次后停止并发送通知
- Q: iCloud API 限流的应对策略？ → A: 自动降低请求频率（降至原速率 50%），持续自适应调整直到限流解除
- Q: 后置处理步骤间的数据传递格式？ → A: 流水线在程序内部处理，步骤间不向外部暴露数据接口，每个步骤对文件进行原地处理或生成新版本
- Q: 视频和 Live Photo 的支持范围？ → A: 支持照片 + 视频 + Live Photo，全部下载，行为与 iCloud 图库一致

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 环境配置与首次同步 (Priority: P1)

作为一名 NAS/服务器用户，我希望通过简单的配置文件设置 iCloud 账号、存储路径和同步参数，系统能自动完成认证并开始第一次照片同步，这样我无需编写代码即可运行服务。

**Why this priority**: 这是系统的入口功能——没有配置和认证，其他所有模块都无法运行。首次同步也是验证整个流水线是否正常工作的最小可行路径。

**Independent Test**: 配置好环境变量文件后，运行容器，系统应完成登录认证（含 MFA）并开始下载 iCloud 照片到指定目录。

**Acceptance Scenarios**:

1. **Given** 用户提供了有效的 iCloud 账号凭证和存储路径配置，**When** 系统启动，**Then** 系统解析配置、初始化运行环境并进入认证流程
2. **Given** 系统已启动并请求认证，**When** 用户在 Telegram Bot 中发送 MFA 验证码，**Then** 认证会话建立并持久化，后续启动无需重新认证（除非会话过期）
3. **Given** 认证已完成，**When** 同步引擎启动，**Then** 系统比对云端与本地照片元数据，仅下载新增或变更的照片

---

### User Story 2 - 后置处理流水线 (Priority: P2)

作为一名高级用户，我希望能对下载完成后的照片进行自定义处理（如格式转换、压缩、添加水印、人脸识别、上传到第三方存储），并且能按顺序组合多个处理步骤。

**Why this priority**: 这是本项目区别于基础 icloudpd 下载器的核心价值——插件化的后置处理让系统从"下载工具"升级为"照片管理入口"。

**Independent Test**: 配置两个后置处理步骤（如先添加水印、再转换为 HEIC 格式），完成照片下载后，系统按顺序执行这些步骤并生成处理后的文件。

**Acceptance Scenarios**:

1. **Given** 用户在配置中定义了一个或多个后置处理任务，**When** 某张照片下载完成，**Then** 触发流水线，按序执行每个处理步骤
2. **Given** 流水线中某个步骤执行失败，**When** 错误发生，**Then** 系统记录失败日志、尝试重试（可配置次数），并继续处理后续步骤（不阻断整条流水线）
3. **Given** 用户未配置任何后置处理，**When** 照片下载完成，**Then** 后置处理流水线被跳过，不影响同步流程

---

### User Story 3 - 消息通知与状态监控 (Priority: P2)

作为一名用户，我希望在同步过程中收到关键事件的通知（如认证失败、同步完成、磁盘空间不足），并能随时查看当前同步状态。

**Why this priority**: 系统通常在无人值守的服务器上运行，用户必须远程感知系统健康状况。没有通知，用户可能在问题发生数天后才察觉。

**Independent Test**: 手动触发一个通知事件（如模拟认证过期），验证通知能通过配置的渠道（如 Webhook、邮件）发送给用户。

**Acceptance Scenarios**:

1. **Given** 用户配置了至少一个通知渠道，**When** 系统发生关键事件（启动、完成、错误、认证过期），**Then** 通知按用户指定的渠道和格式发送
2. **Given** 同步引擎正在运行，**When** 用户查询同步状态，**Then** 系统返回当前进度（已处理照片数、剩余照片数、预计完成时间）
3. **Given** 通知渠道暂时不可用（如网络中断），**When** 系统尝试发送通知失败，**Then** 通知被缓存并在渠道恢复后重试，最多缓存 24 小时后过期丢弃

---

### User Story 4 - 远程控制与监听 (Priority: P3)

作为一名运维人员，我希望不登录宿主机终端就能暂停、恢复或触发同步任务，以及查看系统状态。

**Why this priority**: 通过 Telegram 等即时通讯工具进行远程控制，用户无需登录宿主机或暴露额外端口，既方便又安全。对核心同步功能不是硬依赖。

**Independent Test**: 通过 Telegram Bot 发送"暂停同步"指令，验证同步引擎在下一个检查点暂停。若未配置 Telegram，则通过向本地配置文件写入指令实现相同验证。

**Acceptance Scenarios**:

1. **Given** 远程控制服务已启用，**When** 用户发送"暂停同步"指令，**Then** 同步引擎在当前文件传输完成后暂停，状态变更为"已暂停"
2. **Given** 同步已暂停，**When** 用户发送"恢复同步"指令，**Then** 同步引擎从断点继续
3. **Given** 系统空闲，**When** 用户发送"立即同步"指令，**Then** 同步引擎立即启动一轮完整同步
4. **Given** 远程控制服务收到非法指令，**When** 解析失败，**Then** 系统返回清晰的错误消息，不影响当前运行状态

---

### Edge Cases

- 用户切换删除策略（保留→删除）后，历史已保留的本地副本如何处理？
- 当 iCloud 账号启用了高级数据保护（Advanced Data Protection）时，系统无法下载端到端加密的照片（明确排除，ADP 必须关闭）
- 本地存储空间不足时，同步引擎的行为？（暂停、跳过、通知用户）
- 网络长时间中断后恢复，系统如何处理中断期间的增量变更？
- 多个容器实例同时运行（误操作），如何防止重复下载和元数据冲突？
- iCloud 会话被用户在另一端手动撤销时，系统如何检测并提示重新认证？
- iCloud API 限流触发后，download_delay 是否自动翻倍直至限流解除？还是仅使用固定间隔？
- 同步引擎崩溃后自动重试期间，新到达的远程控制指令（如"暂停"）如何处理？
- 后置处理任务执行时间过长（数小时），是否会阻塞下一轮同步周期？

## Requirements *(mandatory)*

### Functional Requirements

**环境与配置管理模块**

- **FR-001**: System MUST 通过环境变量或配置文件读取所有运行参数（iCloud 凭证、存储路径、同步模式、通知渠道等）
- **FR-002**: System MUST 在启动时验证必填配置项的完整性与格式正确性，对无效配置给出明确错误提示
- **FR-003**: System MUST 支持配置变更后无需重建容器即可生效（热加载或重启自动读取）
- **FR-004**: System MUST 以受限用户权限运行，仅能写入指定的下载目录和缓存目录
- **FR-004a**: System MUST 提供云端删除照片时的本地处理策略配置选项（保留本地副本 / 同步删除 / 移至回收站）

**认证与安全模块**

- **FR-005**: System MUST 支持通过用户名/密码发起 iCloud 登录，并通过 Telegram Bot 接收用户发送的 MFA 验证码完成认证流程
- **FR-006**: System MUST 持久化认证会话状态（如 cookie/session token），在会话有效期内免重复登录。会话文件 MUST 设置严格的文件权限（仅运行用户可读写）
- **FR-007**: System MUST 检测会话过期并自动或提示用户重新认证
- **FR-008**: iCloud 密码和敏感凭证 MUST NOT 以明文形式出现在日志输出中

**核心同步引擎**

- **FR-009**: System MUST 比对 iCloud 云端照片库与本地的元数据差异，识别新增、修改和已删除的照片。对已删除照片的处理策略（保留本地副本 / 同步删除）由用户配置决定
- **FR-010**: System MUST 下载新增或变更的照片、视频和 Live Photo 到用户指定的本地目录，并保持原始的目录结构（可按日期/相册等维度组织）
- **FR-011**: System MUST 支持断点续传——若下载中断，已下载的部分不丢失，恢复后从断点继续。同步进程异常退出时 MUST 自动重启（指数退避：1 分钟 → 5 分钟 → 15 分钟），最多重试 3 次，超限后停止并发送通知
- **FR-012**: System MUST 在同步过程中记录下载进度，支持周期性状态输出
- **FR-012a**: System MUST 支持可配置的下载间隔（`download_delay`）和失败重试等待（`retry_interval`），参考 docker-icloudpd 的固定延迟模式，避免触发 iCloud API 限流

**后置处理流水线**

- **FR-013**: System MUST 支持用户以声明式方式定义后置处理步骤及其执行顺序
- **FR-014**: System MUST 以插件式架构加载后置处理器（Python 模块形式），插件实现约定的接口类并由主进程动态加载，添加新处理器无需改动核心代码
- **FR-015**: System MUST 为每个后置处理步骤提供失败重试机制（重试次数可配置）
- **FR-016**: 单个后置处理步骤的失败 MUST NOT 中断整条流水线或阻塞后续文件下载
- **FR-016a**: System MUST 定义标准的后置处理器接口契约（Python 抽象基类），包含初始化、处理单文件、清理三个生命周期方法，以及配置参数声明和版本标识。流水线在程序内部顺序调用各步骤，每个步骤对文件进行原地处理或生成新版本，步骤间不向外部暴露中间数据

**消息通知总线**

- **FR-017**: System MUST 定义标准化的系统事件类型（启动、完成、错误、认证过期、空间不足等）
- **FR-018**: System MUST 支持至少一种通知输出渠道 Webhook（HTTP POST），可对接 Discord、Slack、企业微信等通用消息平台。同时 MUST 支持 Telegram Bot 作为通知渠道（与远程控制共用 Bot）
- **FR-019**: System MUST 允许用户选择性订阅事件类型和配置通知格式
- **FR-020**: System MUST 在通知渠道不可用时缓存未发送的通知，待恢复后重试

**远程控制与监听服务**

- **FR-021**: System MUST 支持通过 Telegram Bot 接收远程控制指令并返回状态响应；若用户未配置 Telegram，则回退为通过本地配置文件（如指令文件）进行控制
- **FR-022**: 远程控制 MUST 支持的命令至少包括：暂停同步、恢复同步、立即同步、查询状态。通过 Telegram 时支持自然语言命令解析
- **FR-023**: 控制接口 MUST 进行基本的指令合法性校验，拒绝无效或恶意指令
- **FR-024**: System MUST 提供只读的状态查询接口，包含当前同步进度、上次完成时间、错误计数等关键指标

### Key Entities

- **配置 (Configuration)**: 系统运行的所有参数集合，包括 iCloud 凭证、存储路径、同步策略、通知设置、后置处理流水线定义。以文件形式持久化，启动时加载。

- **认证会话 (Auth Session)**: 与 iCloud 服务器的登录状态，包含 session token、过期时间、信任设备信息。持久化到本地文件，通过文件系统权限（chmod 600）保护，仅容器运行用户可读写。

- **媒体资产 (Media Asset)**: iCloud 中的照片、视频、Live Photo 及其元数据，包含唯一标识符、文件名、媒体类型、尺寸、创建时间、修改时间、文件大小。本地存储镜像包含下载状态标记。

- **同步任务 (Sync Task)**: 一次同步周期的工作单元，包含开始时间、状态（运行中/暂停/完成/失败）、已处理资产数、待处理资产数。

- **后置处理步骤 (Post-Processing Step)**: 一个可插拔的处理单元，包含处理器标识、配置参数、执行顺序、重试策略。

- **系统事件 (System Event)**: 运行时发生的关键状态变更，包含事件类型、时间戳、严重级别、描述信息。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户可在 10 分钟内完成从配置文件填写到首次同步启动的全流程
- **SC-002**: 首次同步 100 个媒体资产（照片/视频）的端到端耗时不超过外部网络下载时间的 110%（即系统开销 < 10%）
- **SC-003**: 99% 的 MFA 认证流程在用户单次操作内完成（不出现重复验证）
- **SC-004**: 后置处理流水线在任一插件失败时，不影响其他插件的正常执行（成功插件的执行率维持 100%）
- **SC-005**: 通知消息从事件触达到发送的延迟不超过 30 秒（网络正常条件下）
- **SC-006**: 远程控制指令从发送到系统响应的延迟不超过 5 秒
- **SC-007**: 系统在无人值守模式下连续运行 7 天，无需人工干预（除 MFA 过期重新认证外）
- **SC-008**: 50,000 张照片的全量元数据比对在 5 分钟内完成
- **SC-009**: 同步进程崩溃后，系统在 1 分钟内自动恢复并从中断点继续（首次重试）

## Assumptions

- Apple 高级数据保护 (Advanced Data Protection, ADP) 不受支持——用户必须关闭 ADP 才能使用本系统（与 docker-icloudpd 一致）

- 目标用户拥有基本的 NAS/Docker 使用经验，能编辑配置文件或环境变量
- iCloud 账号已开通 iCloud 照片服务，且有网络连接
- 照片库规模上限为 50,000 张照片/视频（中型家庭图库），超出此规模不保证性能
- 宿主机有足够的存储空间存放下载的照片（系统不在下载前检查存储容量，而是按需报错）


- MFA 交互通过 Telegram Bot 完成：系统向用户 Telegram 发送验证提示，用户回复验证码完成认证；若未配置 Telegram 则回退为日志输出 URL 方式
- 单实例运行——不假设多容器并发场景
- iCloud API 限流策略采用固定延迟模式（参考 docker-icloudpd）：可配置的下载间隔和失败重试等待，而非自适应降速
- 后置处理器以 Python 模块形式实现约定的插件接口，由主进程在内部流水线中顺序调用，不向外部暴露中间数据
